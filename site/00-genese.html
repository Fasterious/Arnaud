1. Diagnostic — le vrai problème

GitLab n’était pas un échec, c’était un révélateur. Tu n’as pas un déficit de compétences. Tu as un problème de compression narrative : trop de matière, dispersée, jamais structurée pour être récupérable sous pression (entretien, 1 h, anglais).

Les 3 IA ont convergé là-dessus, mais aucune n’a nommé le piège central : tu confonds 3 projets qui n’ont pas les mêmes exigences. Tant qu’ils sont confondus, tu ne sais pas pour qui tu écris.

Projet A — Introspection

Tri, sensemaking, mémoire

Audience : toi seul. Pas de mise en forme. Format brut, exploratoire, jetable.

Projet B — Mémoire structurée

Second cerveau, vault

Audience : toi + IA. Markdown discipliné, taggable, réutilisable pour générer du contenu.

Projet C — Vitrine publique

Site, portfolio, narration

Audience : recruteurs, futurs collaborateurs, pairs. 90 s d’attention. Curé, narratif.

Idée-clé

Le projet B (mémoire structurée) est le moteur. Le projet C est une projection filtrée du B. Le projet A est l’input. Ne les confonds plus.

3. Là où je challenge — toi et les IA

Neuf points où la conversation glisse ou manque un angle.

Challenge #1 — Chatbot adaptatif (piège V1–V3)

Un recruteur a 90 secondes et veut scanner, pas dialoguer. L’adaptation par audience : des URLs curées (/for-product-design, /for-research), pas une couche conversationnelle sur la home.

Challenge #2 — « IA dans le process » vs « IA dans le produit »

La maîtrise de l’IA se montre dans comment tu construis le site (case studies, page process, commits) — pas en mettant un chatbot sur la home.

Challenge #3 — « Content first » trop binaire

Tu as raison sur les allers-retours. Il faut une chose figée pour démarrer : un template de case study (frontmatter + 7 sections). Ensuite contenu et forme co-évoluent.

Challenge #4 — Astro vs Next.js

Astro excelle sur le pur statique ; toi tu vis génération côté serveur, traduction, IA plus tard. Next.js 15 + App Router, stack déjà maîtrisée.

Challenge #5 — Obsidian n’est pas « un autre stockage »

C’est une vue sur les mêmes fichiers Markdown du repo (vault/). Source unique, outils multiples.

Challenge #6 — FR + EN en parallèle dès J1

FR = source de vérité ; EN = généré + relu. Champ translation_reviewed dans le frontmatter.

Challenge #7 — « Site = méta case study » en V1

À garder pour V2+. Fais d’abord 3 case studies réelles, puis « How I built this ».

Challenge #8 — Le site ne résout pas GitLab

Le blocage est la fluence sous pression en anglais sur des stories non rejouées. Il faut un track parallèle entretien (§7).

Challenge #9 — Oscillation MVP

MVP (2 semaines) : home + about + 3 case studies + CV + contact. FR seulement. Pas de visu, pas de chatbot. Statique.
Tout le reste : backlog, examen au sprint 3.

4. Approche recommandée

Positionnement

AI-augmented Product Designer / Researcher : recherche → conception → prototype testable → narration. Mantras : Création + Collaboration.

Architecture (chaos privé)

Pas de « garden public » : le chaos reste privé ; le site est la projection curée.

L0 Inbox _inbox/ — dumps vocaux, jetable. Peut être gitignored.
L1 Vault vault/ — Markdown + frontmatter. Moteur. Mémoire réelle.
L2 Site app/ — Next.js, consomme le vault sélectivement. 90 s de lecture.

Repo unique ; vault/ privé ou gitignored si le repo est public. Tu ne publies pas le garden entier.

§B — Reconstruire des métriques absentes

Principal blocage à l’écriture des case studies : la peur de ne pas avoir de résultats chiffrés.

Hiérarchie des preuves (du plus dur au plus mou)

  1. Chiffre direct — conversion, NPS, temps, taux d’erreur, €.
  2. Chiffre indirect — décision prise grâce au livrable, adoption.
  3. Témoignage attribué — citation exacte d’un PM, sponsor, utilisateur.
  4. Comparaison avant/après qualitative — bascule observable.
  5. Ce qui n’est pas arrivé — produit non lancé grâce à une recherche, économies évitées.

Règle : si aucune des 5 ne s’applique, ce n’est probablement pas un case study mais une note.

Patron quand le résultat est faible

La recommandation a été [livrée / publiée / présentée à X].
Elle a été [utilisée / archivée / partiellement reprise] dans [contexte].
À recul, ce que j'en retire de plus utile est [leçon / pattern].

§C — Templates secondaires

V1 ne donne que le template case study. Il en manque 4 pour couvrir tout le vault.

_inbox/<projet>/_template-note-libre.md

Frontmatter minimal : type: note, project, source, status: brut. Aucune section imposée.

vault/fr/skills/_template-skill.md

Champs : category, level, evidence_count. Sections : Définition · Acquisition · Preuves · Ce que je sais ne PAS faire.

vault/fr/reflexions/_template-reflexion.md

Champs : themes, maturity. Sections : Question · Mon hypothèse · Ce qui me ferait changer d’avis.

vault/fr/visions/_template-vision.md

Champs : horizon, confidence. Sections : Ce que je crois · Fondements · Implications.

§E — « Ce que ça dit de moi » : méthode anti-bullshit

Section la plus difficile du template. Protocole en 3 questions à se poser dans l’ordre :

  1. Quel choix particulier as-tu fait qu’un autre n’aurait pas fait ? Pas « j’ai mené des interviews » ; mais « j’ai arrêté à 5 alors qu’on m’en demandait 12, parce que… ».
  2. Qu’est-ce que ça révèle d’une croyance que tu portes ? Pas « je suis pragmatique » ; mais « je crois que matérialiser tôt produit plus d’apprentissage que documenter exhaustivement » — testable, donc crédible.
  3. Lien explicite à un mantra (Création OU Collaboration).

Format final (5 lignes max)

# Ce que ça dit de moi
Ce projet a été un cas typique de [tension récurrente].
J'ai tranché en [choix concret], en assumant [risque].
Ce choix incarne ma croyance que [conviction testable].
C'est aligné avec le mantra **[Création OU Collaboration]** parce que [lien d'une ligne].

Si tu n’arrives pas à remplir ce gabarit, le case study n’est pas mûr pour être publié.

Méta — Workflow couche par couche

Le système se prouve par son existence : ce document est lui-même produit selon la méthode qu’il décrit.

L0 — Inbox privé

Dumps vocaux, idées jetables

_inbox/<projet>/. Aucune contrainte. Gitignored.

L1 — Vault moteur

Markdown discipliné

vault/fr/<projet>/. Frontmatter obligatoire. Versionné.

L2 — Site public

Projection curée

site/<projet>.html aujourd’hui, app/ Next.js plus tard.

Règle d’or — propagation contrôlée

Une couche n’est jamais écrite sans validation explicite sur la précédente.

  • L0 → L1 : l’IA propose un brouillon dans le chat, demande « OK pour écrire dans vault/ ? ».
  • L1 → L2 : l’IA détecte le décalage, montre le diff conceptuel, demande confirmation avant d’écrire dans site/.
  • Nouvelle version : jamais d’écrasement. Créer <slug>-v2.md avec based_on:.

Cohérence inter-couches

  • Chaque L1 déclare ses sources: (L0), son based_on: (L1 précédent), son preview: (L2).
  • Chaque L2 affiche un footer Provenance (ci-dessous).
  • Le fichier racine CHANGELOG.md log toutes les transitions L0→L1 et L1→L2.

Contrôle utilisateur

L’IA n’a jamais le droit, sans demander :

  • d’écraser un .md existant dans vault/ ;
  • de modifier un site/*.html sans avoir d’abord montré le diff conceptuel ;
  • de créer un nouveau projet sans valider le nom avec toi ;
  • de retirer une entrée du backlog.md.
