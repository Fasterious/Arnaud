1. Diagnostic — le vrai problème

GitLab n’était pas un échec, c’était un révélateur. Tu n’as pas un déficit de compétences. Tu as un problème de compression narrative : trop de matière, dispersée, jamais structurée pour être récupérable sous pression (entretien, 1 h, anglais).

Les 3 IA ont convergé là-dessus, mais aucune n’a nommé le piège central : tu confonds 3 projets qui n’ont pas les mêmes exigences. Tant qu’ils sont confondus, tu ne sais pas pour qui tu écris.

Projet A — Introspection

Tri, sensemaking, mémoire

Audience : toi seul. Pas de mise en forme. Format brut, exploratoire, jetable.

Projet B — Mémoire structurée

Second cerveau, vault

Audience : toi + IA. Markdown discipliné, taggable, réutilisable pour générer du contenu.

Projet C — Vitrine publique

Site, portfolio, narration

Audience : recruteurs, futurs collaborateurs, pairs. 90 s d’attention. Curé, narratif.

Idée-clé

Le projet B (mémoire structurée) est le moteur. Le projet C est une projection filtrée du B. Le projet A est l’input. Ne les confonds plus.

3. Là où je challenge — toi et les IA

Neuf points où la conversation glisse ou manque un angle.

Challenge #1 — Chatbot adaptatif (piège V1–V3)

Un recruteur a 90 secondes et veut scanner, pas dialoguer. L’adaptation par audience : des URLs curées (/for-product-design, /for-research), pas une couche conversationnelle sur la home.

Challenge #2 — « IA dans le process » vs « IA dans le produit »

La maîtrise de l’IA se montre dans comment tu construis le site (case studies, page process, commits) — pas en mettant un chatbot sur la home.

Challenge #3 — « Content first » trop binaire

Tu as raison sur les allers-retours. Il faut une chose figée pour démarrer : un template de case study (frontmatter + 7 sections). Ensuite contenu et forme co-évoluent.

Challenge #4 — Astro vs Next.js

Astro excelle sur le pur statique ; toi tu vis génération côté serveur, traduction, IA plus tard. Next.js 15 + App Router, stack déjà maîtrisée.

Challenge #5 — Obsidian n’est pas « un autre stockage »

C’est une vue sur les mêmes fichiers Markdown du repo (vault/). Source unique, outils multiples.

Challenge #6 — FR + EN en parallèle dès J1

FR = source de vérité ; EN = généré + relu. Champ translation_reviewed dans le frontmatter.

Challenge #7 — « Site = méta case study » en V1

À garder pour V2+. Fais d’abord 3 case studies réelles, puis « How I built this ».

Challenge #8 — Le site ne résout pas GitLab

Le blocage est la fluence sous pression en anglais sur des stories non rejouées. Il faut un track parallèle entretien (§7).

Challenge #9 — Oscillation MVP

MVP (2 semaines) : home + about + 3 case studies + CV + contact. FR seulement. Pas de visu, pas de chatbot. Statique.
Tout le reste : backlog, examen au sprint 3.

4. Approche recommandée

Positionnement

AI-augmented Product Designer / Researcher : recherche → conception → prototype testable → narration. Mantras : Création + Collaboration.

Architecture (chaos privé)

Pas de « garden public » : le chaos reste privé ; le site est la projection curée.

L0 Inbox _inbox/ — dumps vocaux, jetable. Peut être gitignored.
L1 Vault vault/ — Markdown + frontmatter. Moteur. Mémoire réelle.
L2 Site app/ — Next.js, consomme le vault sélectivement. 90 s de lecture.

Repo unique ; vault/ privé ou gitignored si le repo est public. Tu ne publies pas le garden entier.

7. Track parallèle (entretien / anglais)

Le site est nécessaire, pas suffisant.

Stories STAR

~30 min / étude

Version parlée 2 min (S-T-A-R) par case study — à mémoriser.

Anglais ciblé

~20 min / jour

Relire tes traductions EN ; reformuler à voix haute.

Mock interview

Avant fin S+4

Entretien blanc en anglais avec un pair UX/PM.

Pourquoi ça compte

Au téléphone, c’est toi qui parles. Le site ouvre la porte ; le track parallèle te fait la franchir sous pression.
